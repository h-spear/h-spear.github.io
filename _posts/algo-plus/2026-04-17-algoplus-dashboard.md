## **대시보드 구성**
---

### **모니터링이 필요했던 이유**
**`Algo Plus`**는 코드 실행을 위해 JDoodle API를 사용하고 있었습니다.  
JDoodle API는 요금제에 따라 일일 호출 가능 횟수가 제한되는데, 서비스 초기에는 하루 500~1,000회 정도의 호출량을 예상하고 약 15,000원(API 하루 1,000회 호출) 수준의 요금제로 시작했습니다.  
이후 실제 사용량을 확인한 결과, 하루 1,000회를 넘는 경우가 많았고, 이에 맞춰 월 약 30,000원(API 하루 2,000회 호출) 수준의 요금제로 상향해 운영하게 되었습니다.  

문제는 이 호출량을 확인하기 위해 **매번 JDoodle 사이트에 직접 접속해야 한다는 점**이었습니다.  
이는 JDoodle만의 문제가 아니었습니다. **`Algo Plus`**는 여러 외부 서비스를 연동하여 운영되고 있었기 때문에, 각 서비스의 상태를 파악하려면 플랫폼마다 일일이 접속해야 했습니다.

- **JDoodle**: 당일 크레딧 소진량 확인을 위해 대시보드 접속
- **AWS Lambda**: 월간 호출 횟수와 무료 할당량 사용량 확인을 위해 AWS 콘솔 접속
- **GitHub**: 기여자 수, PR 등 개발 현황과 이슈 현황 확인을 위해 GitHub 리포지토리 접속
- **크롬 웹 스토어**: 사용자 수 및 리뷰(버그 제보 등) 확인을 위해 스토어 접속

<span style="background-color:yellow">**각 정보를 확인하기 위해 여러 사이트를 오가는 과정은 번거로웠고, 서비스 상태를 한눈에 파악하기도 어려웠습니다.**</span>  
이러한 불편함을 해소하기 위해 **흩어진 정보를 한 곳에 모아 관리할 수 있는 내부 운영 대시보드**를 직접 구축하기로 했습니다.

<br/>

### **수집할 데이터 정의**

대시보드를 만들기 전, 가장 먼저 한 일은 **"어떤 정보를 한눈에 봐야 하는가"**를 정의하는 것이었습니다.  
개발 현황, 비용 및 서비스 안정성, 사용 현황의 세 가지 관점에서 각 플랫폼별 수집 항목을 정리했습니다.

| **서비스** | **수집 정보** | **관점** |
|-----------|-------------|---------|
| **GitHub** | 버전, 기여자 수, 브랜치/커밋/스타/포크/PR 수, 버그 이슈 목록 | 개발 현황 |
| **JDoodle API** | 당일 크레딧 사용량(매일 9시 초기화) / API 응답 시간 및 정상 여부 | 비용 및 서비스 안정성 |
| **AWS Lambda** | 월 누적 호출 수 / API 게이트웨이 응답 시간 및 정상 여부 | 비용 및 서비스 안정성 |
| **크롬 웹 스토어** | 사용자 수, 리뷰 수, 평점 | 사용 현황 |

한 가지 아쉬운 점은, **백준의 제출(Submit) API를 Health Check 대상에 포함하고 싶었지만 구현하지 못했다는 것**입니다.  

**`Algo Plus`**는 백준 문제 풀이 화면에 웹 IDE를 얹는 구조이기 때문에, 제출 API의 정상 동작 여부는 서비스 동작에 직접적인 영향을 미쳤습니다.  
하지만 백준의 제출 API는 저희가 직접 제어할 수 있는 방법이 없었고, 안정적으로 상태를 확인할 방법도 없어서 모니터링 대상에 포함하기 어려웠습니다.  

결국 `Algo Plus`의 백준 제출 기능은 수동으로 직접 확인할 수밖에 없었습니다.  
실제로 운영 중에는 페이지 구조나 API 형식 변경으로 기존 코드가 동작하지 않는 버그가 발생하기도 했습니다.

### **아키텍처 설계**

기존 **`Algo Plus`**의 서버리스 구조를 그대로 유지하면서, 추가 서버 없이 모니터링 시스템을 구축했습니다.

<img src="/assets/img/post/algo-plus/dashboard-architecture.png" alt="대시보드 아키텍처" style="max-width: 100%; height: auto;">

- **AWS Lambda**: 각 플랫폼에서 데이터를 수집하고 Firestore에 저장. 기존 컴파일 Lambda와 별개 함수로 분리
  - **EventBridge**: 주기적으로 모니터링을 위한 AWS Lambda 함수 실행(1시간에 1번)
- **Firebase Firestore**: 백엔드 서버 없이 Firebase SDK로 대시보드에서 직접 조회 가능
- **React Dashboard**: 프론트엔드만 구현하여 GitHub Pages로 배포

## **대시보드가 보여준 것**
---

아래 차트는 대시보드 구축 시점(2025.06.21)부터 서비스 종료(2026.02.28)까지의 API 호출량과 사용자 수 추이입니다.  
주황선은 **하루 API 호출량(단위: 회)**, 녹색 막대는 **사용자 수(단위: 명)**를 의미합니다.  

<img src="/assets/img/post/algo-plus/dashboard-chart.png" alt="API 사용량 차트" style="max-width: 100%; height: auto;">

> 차트 중간(2025년 9월)에 데이터가 수집되지 않은 구간이 있는데, 해당 시점에 대시보드 개선 작업이 진행되면서 Lambda 수집 함수가 일시적으로 중단됐기 때문입니다.

차트를 보면 **2025년 7월 초** 호출량이 최고치(약 1,600회), **2025년 10월 초**에 두 번째 피크를 찍었지만, 이후로는 이전 수준으로 회복되지 않았습니다.
**`Algo Plus`**는 SSAFY 내에서 개발/홍보가 이루어졌기 때문에, 방학이나 알고리즘 집중 교육 기간에 사용량이 올랐던 것으로 보입니다.  

가장 중요한 건 <span style="background-color:yellow">**사용자 수는 완만하게 유지되는 반면, 실제 호출량은 꾸준히 감소**</span>했다는 점입니다.  
**설치는 되어 있지만 실제로 쓰지 않는 사용자**가 늘어나고 있다는 의미였습니다.

**2026년 초**에는 호출량이 대부분 200회 이하로 떨어졌습니다.  
사용량 감소에 맞춰 요금제를 월 15,000원(하루 1,000회)으로 낮추기도 했지만, 그마저도 한도를 채우는 날이 거의 없었습니다.  
이 데이터가 팀원들과 서비스 방향성을 논의하는 결정적인 근거가 되었습니다.

<br/>

## **서비스 종료 결정**
---

**`Algo Plus`**는 약 1년간 수익 모델 없이, 순수하게 사용자에게 도움이 되고자 팀원들의 애정으로 운영된 서비스입니다.  
그 과정에서 JDoodle API 요금제(월 약 30,000원)는 팀원들이 사비로 분담하며 유지해왔습니다.

그러나 대시보드를 통해 확인한 데이터는 기대와 달랐습니다.  
<span style="background-color:yellow">**사용량은 꾸준히 줄어드는데 고정 비용은 그대로였고, 팀원들이 사비로 API 비용을 부담하는 것이 점점 비효율적이라는 판단에 이르게 되었습니다.**</span>

감소하는 사용량과 지속되는 비용 부담, 두 가지 요인이 겹치면서 팀원들과의 논의 끝에 **`Algo Plus`** 서비스를 종료하기로 결정했습니다.

## **마무리 회고**
---

**`Algo Plus`**는 저에게 서비스 배포 경험을 처음 안겨준 프로젝트였습니다.  
아이디어를 구체화하는 기획 단계부터, 크롬 익스텐션 개발과 AWS 서버리스 구조 설계, 스토어 배포와 사용자 피드백 대응, 그리고 모니터링 시스템 구축과 서비스 종료까지 경험할 수 있었습니다.  
사이드 프로젝트임에도 불구하고 서비스를 운영하는 과정에서 배울 수 있는 것들이 많았습니다.  

그중에서도 모니터링 대시보드 구축 경험은 특히 기억에 남습니다.  
여러 플랫폼을 직접 접속해서 확인해야 한다는 불편함을 해결하기 위해 필요한 정보를 한 곳에 모아 직접 시각화한 경험이었습니다.  
그리고 그 데이터가 단순한 참고 자료에 그치지 않고, **팀의 방향성을 결정하는 실질적인 근거**가 되었다는 점에서 의미가 깊게 남았습니다.  

최근 백준도 서비스 종료 소식을 알렸습니다.  
같이 개발했던 팀원들은 백준에서 알고리즘 문제 푸는 것을 좋아해서 **`Algo Plus`**도 만들었고 이후에는 애정으로 꾸준히 유지해 왔습니다.  
백준 역시 4월 28일부로 서비스를 종료한다는 소식을 들었을 때, 마음이 아쉬웠습니다.  

마지막으로, 함께 서비스를 만들고 운영해온 팀원들, **`Algo Plus`**를 사용해준 사용자들, 그리고 **출발점이 되어준 백준**에 감사를 전하며, **`Algo Plus`**에 대한 이야기를 여기서 마무리합니다.
