이 문제는 '최소 회의실 개수' 문제로, 여러 회의의 시작과 끝 시간이 주어졌을 때, 모든 회의를 겹치지 않고 진행하려면 최소 몇 개의 회의실이 필요한지를 구하는 문제였습니다. 저는 **누적 합**을 활용해 문제를 해결했지만, 스터디원들은 **그리디 + 우선순위 큐**로 접근했습니다. 인터넷에서 검색해보니 대부분의 풀이가 우선순위 큐를 사용하는 방식이었습니다.

해당 문제에 대해 서로의 풀이법을 공유하고 의견을 주고받았던 경험이 특히 기억에 남습니다. 다양한 접근법이 가능하다는 점을 공유하면 좋을 거 같아 이번에 아이디어와 구현 방법을 정리해 공유해봅니다.

## **접근법1: 그리디 + 우선순위 큐**

### **아이디어**

핵심 아이디어는 **시작 시간이 빠른 회의**부터 순차적으로 탐색을 하면서 **<span style="background-color:yellow">현재까지 생성한 회의실 중</span> 비어있는 회의실이 있다면** 해당 회의실에서 회의를 진행하고, **비어있는 회의실이 없다면 새로운 회의실을 생성**하는 것입니다.

해당 입력에 대해서는 3이 정답입니다.  

### **우선순위 큐 적용**

다만 지금까지 생성된 회의실을 일일이 살펴보면서, 종료 시간을 확인하는 것은 비효율적입니다. 이 방식을 사용한다면 하나의 회의를 배정할 때마다, 회의실을 탐색하는 시간 복잡도는 $O(N)$입니다.  

사실 우리는 모든 회의실을 탐색할 필요 없이, 최대한 종료 시간이 빠른 회의실만 찾아 배정하면 됩니다.(Greedy)  
그러기 위한 자료구조로는 **우선순위 큐**를 사용합니다. 우선순위 큐를 **종료 시간**을 기준으로 관리하면, 가장 빨리 비는 회의실을 효율적으로 추적할 수 있습니다.  

우선순위 큐의 시간 복잡도는 $O(log N)$이기 때문에, 우리는 $O(log N)$시간 만에 현재 회의실 중 가장 빨리 비는 회의실을 찾아 배정할 수 있습니다.

### **시간 복잡도**
- $O(N \log N)$

  - 데이터 입력 : $O(N)$
  - 정렬 : $O(N \log N)$
  - 회의실 배정 : $O(N \log K)$ (K: 현재 우선순위 큐에 들어있는 회의실 수, 최악의 경우 N)

### **전체 코드(Java)**

회의실 배정 로직에서 `heap.poll()`하고, `heap.add()`하는 부분이, 이미 생성된 회의실을 재사용하여 현재 회의를 배정하는 역할을 합니다.  
만약 `heap.poll()`을 수행하지 않고 바로 `heap.add()`만 수행된다면, 이는 **배정할 수 있는 회의실이 없어 새로운 회의실을 생성하고 해당 회의를 배정하는 경우로 볼 수 있습니다.**

최종적으로 우선순위 큐에 들어있는 원소의 개수가 **최소 회의실 개수**가 됩니다.

```java
import java.io.*;
import java.util.*;

public class Main {

	public static void main(String[] args) throws Exception {
		BufferedReader br = new BufferedReader(new InputStreamReader(System.in));
		StringTokenizer st;

		// 입력
		int N = Integer.parseInt(br.readLine());
		Meeting[] meetings = new Meeting[N];
		for (int i = 0; i < N; ++i) {
			st = new StringTokenizer(br.readLine());
			meetings[i] = new Meeting(
				Integer.parseInt(st.nextToken()),
				Integer.parseInt(st.nextToken())
			);
		}

		// 회의 정렬
		Arrays.sort(meetings, (o1, o2) -> Integer.compare(o1.start, o2.start));

		// 우선순위 큐
		Queue<Integer> heap = new PriorityQueue<>();

		// 회의실 배정
		for (Meeting meeting: meetings) {
			if (!heap.isEmpty() && heap.peek() <= meeting.start) {
				heap.poll();
			}
			heap.add(meeting.end);
		}

		// 출력
		System.out.println(heap.size());
		br.close();
	}

	private static class Meeting {
		int start, end;

		Meeting(int start, int end) {
			this.start = start;
			this.end = end;
		}
	}
}

```

## **접근법2: 누적합**
---

### **브루트포스**
저는 알고리즘 문제를 풀 때 우선 브루트포스 해법으로 시간 복잡도를 따져보고, 만약 제한을 초과하면 더 효율적인 알고리즘을 적용할 수 있는지 고민합니다.   
이번 문제 역시 처음 접했을 때 가장 먼저 단순한 브루트포스 풀이를 떠올렸습니다.   
즉, 모든 회의마다 for문으로 시간 배열에 회의실의 수를 카운팅하는 방식입니다.   

시간 배열에 카운팅을 완료한 후, **<span style="background-color:yellow">배열의 최댓값이 필요한 최소 회의실 개수</span>**가 됩니다.
```java
int[] time = new int[(2 << 31) - 1];
for (Meeting meeting: meetings) {
	for (int i = meeting.start; i <= meeting.end; ++i) {
		++time[i];
	}
}
```

하지만 회의실의 개수가 $100,000$개이고, 시간이 최대 $2^{31}-1$에 이를 수 있기 때문에, 이 방법은 공간과 시간 복잡도 측면에서 매우 비효율적입니다.   
최악의 경우 시간 복잡도가 $O(N \cdot (2^{31} - 1))$이 되기 때문입니다.   
![BOJ_19598_Solution2_Draw](/assets/img/boj/19598/sol2-001.png)

이러한 비효율을 해결하기 위해, 다음 알고리즘을 적용하여 문제를 해결했습니다.

### **누적합**
첫 번째로 개선할 수 있는 부분은 **시간 배열**을 체크하는 부분입니다.   
for문을 사용하지 않고 누적합을 이용하면, 여러 구간에 대한 변화량을 한 번에 기록해두고, 한 번의 순회로 각 시점의 최종 값을 빠르게 계산할 수 있습니다.   

우선 각 회의마다 시작 시간 인덱스에는 +1, 종료 시간 인덱스에는 -1을 더하는 것으로 표시를 해둡니다.   
![BOJ_19598_Solution2_Draw](/assets/img/boj/19598/sol2-002.png)

그리고 왼쪽에서 오른쪽으로 누적을 하면서 더해가면, 각 시점마다 동시에 사용 중인 회의실의 개수를 빠르게 계산할 수 있습니다.   
![BOJ_19598_Solution2_Draw](/assets/img/boj/19598/sol2-003.png)

코드로 표현하면 다음과 같이 됩니다.
```java
// 시간 배열
int[] time = new int[MAX_TIME];

/**
 * 시작 시간 인덱스에 +1
 * (종료 시간 + 1) 인덱스에 -1
 */
for (Meeting meeting: meetings) {
	++time[meeting.start];
	--time[meeting.end];
}

// 누적합
for (int i = 0; i < MAX_TIME - 1; ++i) {
	time[i + 1] += time[i];
}
```

### **좌표 압축**

누적합을 사용해서 시간 복잡도를 많이 개선할 수 있었지만, 여전히 문제가 남아있습니다.   
시간은 최대 $2^{31}-1$이 되기 때문에 해당 크기만큼 선언할 메모리 공간이 부족합니다.   
추가적으로, 한 번의 for문만 수행한다고 해도 $2^{31}-1$이라는 값 자체가 너무 커서 현실적으로 처리하기 어렵습니다.

여기서 중요한 점은, 모든 시간에 대해 배열을 만들 필요가 없다는 것입니다.
예를 들어 [(0, 10), (4, 12)]라는 데이터와 [(0, 2), (1, 3)] 데이터처럼 구간의 상대적 위치만 같으면, 실제로 필요한 최소 회의실 개수는 동일하게 계산됩니다.   
![BOJ_19598_Solution2_Draw](/assets/img/boj/19598/sol2-004.png) 

회의의 개수는 최대 $100,000$개이므로, 좌표 압축을 적용하면 모든 회의의 시간 값은 $200,000$ 이하로 제한할 수 있습니다.   
즉, 시간 배열의 크기를 약 $200,000$ 정도만 선언해도 충분히 문제를 해결할 수 있습니다.

아래는 좌표 압축의 한 가지 구현 예시입니다. 이 방법 외에도 여러 방식으로 좌표 압축을 구현할 수 있습니다.
```java
static void coordinateCompression(Pair[] pairs) {
	Set<Integer> coordSet = new TreeSet<>();
	for (Pair p: pairs) {
		coordSet.add(p.x);
		coordSet.add(p.y);
	}

	List<Integer> sortedList = new ArrayList<>(coordSet);

	Map<Integer, Integer> indices = new HashMap<>();
	for (int i = 0; i < sortedList.size(); i++) {
		indices.put(sortedList.get(i), i);
	}

	for (Pair p: pairs) {
		p.x = indices.get(p.x);
		p.y = indices.get(p.y);
	}
}
```

### **시간 복잡도**

- $O(N \log N)$

  - 데이터 입력 : $O(N)$
  - 좌표 압축 : $O(N \log N)$
  - 누적합 : $O(N)$
  - 최댓값 찾기 : $O(N)$


### **전체 코드(Java)**

```java
import java.io.*;
import java.util.*;

public class Main {

	public static void main(String[] args) throws Exception {
		BufferedReader br = new BufferedReader(new InputStreamReader(System.in));
		StringTokenizer st;

		// 입력
		int N = Integer.parseInt(br.readLine());
		Meeting[] meetings = new Meeting[N];
		for (int i = 0; i < N; ++i) {
			st = new StringTokenizer(br.readLine());
			meetings[i] = new Meeting(
				Integer.parseInt(st.nextToken()),
				Integer.parseInt(st.nextToken())
			);
		}

		// 좌표 압축
		coordinateCompression(meetings);

		// 회의 시간 표시
		int[] time = new int[222222];
		for (Meeting meeting: meetings) {
			time[meeting.start] += 1;
			time[meeting.end] -= 1;
		}

		// 누적합
		for (int i = 0; i < 222221; ++i) {
			time[i + 1] += time[i];
		}

		// 최소 회의실 개수 계산
		int answer = 0;
		for (int i = 0; i < 222222; ++i) {
			answer = Math.max(answer, time[i]);
		}

		// 출력
		System.out.println(answer);
		br.close();
	}

	private static void coordinateCompression(Meeting[] meetings) {
		Set<Integer> coordSet = new TreeSet<>();
		for (Meeting p: meetings) {
			coordSet.add(p.start);
			coordSet.add(p.end);
		}

		List<Integer> sortedList = new ArrayList<>(coordSet);

		Map<Integer, Integer> indices = new HashMap<>();
		for (int i = 0; i < sortedList.size(); i++) {
			indices.put(sortedList.get(i), i);
		}

		for (Meeting meeting: meetings) {
			meeting.start = indices.get(meeting.start);
			meeting.end = indices.get(meeting.end);
		}
	}

	private static class Meeting {
		int start, end;

		Meeting(int start, int end) {
			this.start = start;
			this.end = end;
		}
	}
}

```

### **다른 사람의 풀이**

추가로, 다른 사람이 작성한 풀이도 함께 소개하고자 합니다.  
이 코드는 제가 구현했던 누적합 방식의 코드보다 한 단계 더 개선된 버전이라고 생각합니다.   

제가 작성한 누적합 풀이는 전체 시간 배열을 선언한 뒤, 각 구간별 변화량을 기록하고 누적합을 통해 동시에 사용 중인 회의실 개수를 계산하는 방식이었습니다.   
반면, 이 코드는 각 회의의 시작과 끝 시점에 +1, -1을 기록한 뒤, 정렬해 누적합을 계산합니다.   
불필요하게 큰 배열을 선언하거나 좌표 압축을 하지 않고, 실제 변화가 발생하는 시점만 효율적으로 처리하기 때문에 메모리와 속도 면에서 더욱 최적화된 풀이라고 생각합니다.

한 가지 주의할 점은, **정렬할 때 시간 값이 같다면 -1(회의 종료)이 +1(회의 시작)보다 먼저 오도록 정렬**해야 한다는 점입니다.
이렇게 해야 동시에 시작과 끝이 겹치는 회의가 있을 때, 회의실 개수가 정확하게 계산됩니다.

```java
import java.io.*;
import java.util.*;

public class Main {

	public static void main(String[] args) throws Exception {
		BufferedReader br = new BufferedReader(new InputStreamReader(System.in));
		StringTokenizer st;

		int N = Integer.parseInt(br.readLine());

		List<Pair> events = new ArrayList<>();
		for (int i = 0; i < N; ++i) {
			st = new StringTokenizer(br.readLine());
			int start = Integer.parseInt(st.nextToken());
			int end = Integer.parseInt(st.nextToken());
			events.add(new Pair(start, 1));
			events.add(new Pair(end, -1));
		}

		events.sort((o1, o2) -> {
			if (o1.first == o2.first)
				return Integer.compare(o1.second, o2.second);
			return Integer.compare(o1.first, o2.first);
		});

		int answer = 0;
		int count = 0;
		for (Pair event: events) {
			count += event.second;
			answer = Math.max(answer, count);
		}
		System.out.println(answer);
		br.close();
	}

	private static class Pair {
		int first, second;

		Pair(int first, int second) {
			this.first = first;
			this.second = second;
		}
	}
}
```
