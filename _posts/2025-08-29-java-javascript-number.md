## **배경**
상세한 개발 내용은 생략하지만, 저는 **클라이언트 측에서 전송되어 들어온 JSON 형식의 페이로드가 올바른 타입으로 전달되는지 검증하는 기능**을 개발했습니다.  
페이로드 검증 알고리즘을 작성한 후, 다양한 테스트 케이스를 통해 알고리즘 동작을 확인했습니다.  

먼저 스프링 JUnit으로 테스트 코드를 작성해 알고리즘이 정상적으로 통과하는 것을 확인했고, 이후 실제 서비스 환경에 적용하여 동일한 데이터로 테스트를 진행했습니다.  
그러나 실제 서비스에서 호출 시 일부 테스트 케이스가 실패하는 현상을 발견했습니다.  

### **실패한 테스트 케이스**
서비스 환경에서 실패한 테스트 케이스는 `Long` 타입 필드가 포함된 페이로드였으며, 특히 필드 값으로 `9,223,372,036,854,775,807`을 전달하는 경우에만 실패했습니다.  
이 수치는 자바에서 **Long.MAX_VALUE**에 해당하는 값으로, `Long` 타입 범위 내이기 때문에 제가 개발한 환경(Java, Spring)에서는 타입 불일치가 발생하지 않아야 하는데도 오류가 발생했습니다.  
```json
{
    "field1": {
        "type": "Integer",
        "value": 123456789
    },
    "field2": {
        "type": "Long",
        "value": 92239223372036854775807  <--type_mismatch
    },
    "field3": {
        "type": "String",
        "value": "this is string"
    }
}
```

로그를 찍어 확인해보니, 클라이언트 측에서 `9,223,372,036,854,775,807` 값을 전달했음에도 서버 측에서는 `9,223,372,036,854,776,000` 값이 넘어오는 것을 확인할 수 있었습니다.  
이 값은 `Long` 타입 범위를 벗어나기 때문에 형 변환에 실패하여 오류가 발생했던 것입니다.  
![java_javascript_number_1](/assets/img/post/java-javascript-number/img1.png)   

### **테스트 환경의 차이점**
동일한 알고리즘과 동일한 데이터를 사용했음에도 결과가 달랐던 이유는 테스트 환경의 차이로 추정할 수 있습니다.  
제가 진행한 단위 테스트(JUnit)는 모든 코드가 Java로만 이루어진 환경이었지만, 실제 서비스는 웹 서비스로서 프론트엔드(JavaScript)에서 백엔드(Java, Spring)로 데이터를 전송하여 API를 호출하는 구조였습니다.  

||Component Test|Integration Test|
|------|---|---|
|목적|모듈/함수 단위 검증|시스템 연동 및 데이터 흐름 검증|
|툴|JUnit|수동 테스트|
|언어|Java|JavaScript, Java|
|데이터 입력 방식|코드 내 직접 생성|프론트엔드 전송 데이터 사용(API호출)|
|의존성|외부 의존성 없음|실제 웹 서비스 환경 의존|

## **숫자 타입 한계와 해결 방법**

발생한 문제는 다음과 같습니다.  

> 클라이언트 측(JavaScript)에서 JSON 데이터의 필드 값으로 `9,223,372,036,854,775,807`을 전달하면 서버 측(Java)에서는 `9,223,372,036,854,776,000`으로 변경된다.

### **JavaScript Number의 한계**

이 문제의 근본적인 원인은 JavaScript의 `Number` 타입과 Java의 `Long` 타입 간 표현 가능한 숫자 범위와 정밀도 차이에 있습니다.  

MDN의 JavaScript `Number` 문서에 따르면: 

> JavaScript의 `Number` 타입은 Java 혹은 C#의 `double` 타입처럼 IEEE 754 64비트 바이너리 배정 밀도 값입니다.  

  - 출처: <a href="https://developer.mozilla.org/ko/docs/Web/JavaScript/Reference/Global_Objects/Number" target="_blank">MDN JavaScript Number</a>

또한 JavaScript의 `Number`에는 다음과 같은 Property들이 존재합니다.

- `Number.MAX_SAFE_INTEGER` : JavaScript에서 안전한 최대 정수. ( $2^{53}-1$ )
- `Number.MIN_SAFE_INTEGER` : JavaScript에서 안전한 최소 정수. ( $-(2^{53}-1)$ )

해당 문서를 통해 알 수 있듯, JavaScript의 `Number`는 부동소수점 형식을 사용하며, 안전하게 표현할 수 있는 정수 범위가 제한되어 있습니다.

**따라서 JavaScript에서는 `9,223,372,036,854,775,807`($2^{63}-1$)과 같은 값을 정확하게 표현할 수 없어, <span style="background-color:yellow">근사값으로 처리하는 바람에 서버에 데이터가 변형되어 전달된 것입니다.</span>**  

이 사실이 신기하여 웹 브라우저 콘솔에서 바로 $2^{53}-1$을 초과하는 값을 직접 출력해보니, 값이 달라지는 것을 확인할 수 있었습니다.
```javascript
console.log(9223372036854775807);   // 출력: 9223372036854776000
console.log(9007199254740993);      // 출력: 9007199254740992
console.log(900123412349999999);    // 출력: 900123412350000000
```

심지어 동등 비교 연산자 결과도 다음과 같았습니다.
```javascript
console.log(9223372036854775807 === 9223372036854776000);   // 출력: true
console.log(9007199254740993 === 9007199254740992);         // 출력: true
console.log(900123412349999999 === 900123412350000000);     // 출력: true
```

#### **Java Long**
`Long` 타입은 64비트 정수형으로, 부호 있는 2의 보수법을 사용해 숫자를 표현합니다.  
표현 가능한 정수 범위는 `-9,223,372,036,854,775,808`부터 
`9,223,372,036,854,775,807`까지이며, 이 범위 내의 값을 정확하게 표현할 수 있습니다.


### **BigInt vs. String**

#### **1. BigInt 활용**

찾아보니 JavaScript에는 `BigInt` 객체가 있습니다.  

> `BigInt`는 `Number` 원시 값이 안정적으로 나타낼 수 있는 최대치인 2^53 - 1보다 큰 정수를 표현할 수 있는 내장 객체입니다.

  - 출처: <a href="https://developer.mozilla.org/ko/docs/Web/JavaScript/Reference/Global_Objects/BigInt" target="_blank">MDN JavaScript BigInt</a>

이 객체를 사용하면, `Number` 타입으로는 표현할 수 없었던 큰 숫자도 정확하게 표현하고 연산할 수 있습니다.  
하지만 `Number`와는 별개의 다른 타입이기 때문에 혼용 시 코드가 복잡해질 수 있으며, JSON 직렬화가 어렵다는 문제가 있습니다.  
특히 `JSON.stringify()`를 바로 사용할 수 없습니다.

#### **2. 문자열 활용**

큰 숫자를 다룰 때 또 한가지 방법은 숫자를 문자열 형태로 전달하는 것입니다.  
클라이언트에서 매우 큰 숫자를 문자열로 변환해 전송하면, 숫자 정밀도 손실 없이 값을 안전하게 전달할 수 있습니다.  
또한, 이 방법은 **JSON 직렬화 문제 없이 데이터를 주고받을 수 있어** 호환성이 뛰어납니다.
```json
{
    "field2": {
        "type": "Long",
        "value": "92239223372036854775807"
    },
}
```

#### **3. 요구사항에 맞는 해결책 선택**
제가 개발하고자 하는 요구사항에서 해당 문제를 해결하기 위한 가장 적합한 방법은 **String 활용** 방식입니다.  
이유는 다음과 같습니다.  
1. **클라이언트 측에서 큰 수에 대한 연산이 불필요함**
2. **JSON 직렬화가 용이함**

제가 개발하는 기능에서 요구사항은 단순히 JSON 형식의 데이터를 전송하는 것이므로, 직렬화가 어려운 `BigInt` 보다 문자열 방식을 사용하는 것이 적합합니다.  
반면, 클라이언트 측에서 큰 수에 대한 연산이 필요한 요구사항이라면 `BigInt` 사용을 고려할 수 있습니다.

||BigInt|문자열(String)|
|------|---|---|
|표현 범위|임의 정밀도 정수 표현|무제한(문자열 길이만큼 표현)|
|숫자 연산|직접 연산 가능|직접 연산 불가능(변환 후 연산 필요)|
|JSON 직렬화|기본 직렬화 불가(별도 처리 필요)|기본 문자열로 직렬화 가능|

## **배운 점**

**1. JavaScript와 Java 간 숫자 처리 방식의 근본적인 차이**  
JavaScript의 `Number` 타입은 부동소수점 형식으로 표현되며, 안전하게 표현할 수 있는 정수 범위가 제한적이라는 사실을 알게 되었습니다.  
특히 이 범위가 자바의 `Long` 타입보다 작기 때문에, 큰 숫자 데이터를 전송할 때는 문자열로 처리해야 데이터 손상을 방지할 수 있다는 점을 배웠습니다.  

**2. 테스트의 중요성**  
단위 테스트 환경과 실제 서비스 환경에서 테스트 결과가 달라지는 것을 몸소 경험하였습니다.  
웹 개발에서 클라이언트-서버 간 통신뿐만 아니라, 서로 다른 시스템 간 연동 시 각 시스템의 차이로 인해 예상치 못한 오류가 발생하고, 이것이 서비스 장애로 이어질 수 있음을 깨달았습니다.  
이러한 문제를 예방하기 위해, 개발 과정에서 작성한 로직에 대해 단위 테스트뿐만 아니라 통합 테스트 등 다양한 검증 절차를 수행해야 한다는 점을 다시 한번 느꼈습니다.
